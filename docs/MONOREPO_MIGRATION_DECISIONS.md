## Purpose

The repo audit (Phase 2) surfaced 10 open questions before the monorepo shell can be
created safely. This document resolves those questions as explicit decisions so future
sessions and AI assistants do not re-litigate them.

Each decision records what was decided, why, and what it requires before proceeding.

---

## Decision 1 — Framework Strategy

**Decision: use a mixed-framework monorepo at first.**

GuideForge remains:
- Next.js 16.2.4 App Router
- React 19
- Server-side API routes for AI generation

Techsperts remains:
- Vite 6.3.5 SPA
- React Router DOM v7
- React 18.3.1
- Supabase edge functions (Deno) for AI generation

**Why:** Migrating Techsperts from Vite to Next.js is a separate major project. Doing it
at the same time as the monorepo move would combine two high-risk changes and multiply
the blast radius if something breaks. Vite and Next.js can coexist in a pnpm workspace.

**Policy:**
- The first monorepo shell must support both apps as-is.
- pnpm workspace treats each app as an independent build unit; both can coexist.
- The Vercel deployment for each app continues to use its own build system.
- A future Techsperts Next.js migration can be planned only after both apps are stable
  in the monorepo and both deploy cleanly from `apps/*`.
- That future migration is a separate decision and a separate bundle, not part of the
  initial monorepo move.

**Resolves:** Open Question #1 from the audit.

## Decision 2 — React Version Strategy

**Decision: do not force React version alignment during the initial monorepo move.**

GuideForge stays on React 19. Techsperts stays on React 18.3.1.

**Why:** Both apps work correctly on their current React versions. Forcing alignment during
the same session as the monorepo move adds unnecessary risk. If either upgrade breaks a
runtime behavior, it becomes hard to isolate whether the monorepo move or the React upgrade
caused it.

**Policy:**
- Early shared packages must be pure TypeScript and framework-neutral. They must not
  import React, ReactDOM, or any React-specific hooks.
- `packages/shared-types`, `packages/domain-templates`, `packages/core`, and the
  type/contract layers of `packages/ai` and `packages/governance` should have no React
  dependency.
- `packages/design-system` must be deferred until after both apps are stable in the
  monorepo, because shared UI components create React peer-dependency version conflicts.
- React version alignment is a separate future decision. When it happens, it must be its
  own dedicated bundle with typecheck + build + manual test verification for each app.

**Resolves:** Open Question #2 from the audit.

## Decision 3 — Package Manager Strategy

**Decision: use pnpm workspaces as the monorepo package manager.**

**Why:** Both GuideForge and Techsperts already use pnpm and have `pnpm-lock.yaml` files.
pnpm workspaces are the lowest-friction option — no migration of lockfiles, no new tooling,
no relearning.

**Policy:**
- Root `pnpm-workspace.yaml` should declare:
  ```yaml
  packages:
    - 'apps/*'
    - 'packages/*'
  ```
- Root `package.json` should be `"private": true` with no dependencies — workspace root
  only.
- App `package.json` files remain in place initially and are not modified during the first
  move.
- Turborepo (`turbo.json`) is optional and deferred. Do not add it in the initial shell.
  It can be added later when there is a real need for pipeline orchestration.
- Do not aggressively hoist shared dependencies during the first move. Let each app manage
  its own `node_modules` initially.

**Resolves:** Open Question #3 (TypeScript setup) is addressed by Decision 2 — Techsperts'
TypeScript setup via Vite passthrough is acceptable for now and does not need a formal
`tsconfig.json` before the monorepo move.

## Decision 4 — Techsperts Supabase Env Cleanup

**Decision: replace the hardcoded Supabase info pattern in Techsperts before moving it
into the monorepo.**

The audit found `utils/supabase/info.tsx` is an autogenerated file with hardcoded
`projectId` and `publicAnonKey`. This pattern must be replaced with env vars before the
monorepo move so that the monorepo never contains committed credentials.

**Why:** A monorepo is more likely to have contributors and AI assistants reading across
its full history. Committed credentials — even public anon keys — are bad hygiene and can
become harder to rotate if they ever appear in a shared repo's git history.

**Required env var names (names only — values must stay in .env.local and Vercel):**

| Env var | Usage | Who uses it |
|---------|-------|-------------|
| `VITE_SUPABASE_URL` | Supabase project URL (already used in generateProvisionalGuide.ts) | Vite client |
| `VITE_SUPABASE_ANON_KEY` | Supabase public anon key (currently hardcoded) | Vite client |

The `utils/supabase/client.tsx` currently derives the URL from `projectId` in info.tsx.
After cleanup, it should read from `import.meta.env.VITE_SUPABASE_URL` and
`import.meta.env.VITE_SUPABASE_ANON_KEY` directly.

**Do not print or commit actual key values anywhere. Names only in this doc.**

**Cleanup scope (dedicated pre-migration bundle):**
1. Create or verify `.env.local` in Techsperts with the required vars.
2. Update `utils/supabase/info.tsx` or replace it so the client reads from env vars.
3. Update `utils/supabase/client.tsx` to not depend on hardcoded values.
4. Verify Supabase auth still works after the change.
5. Add `utils/supabase/info.tsx` to `.gitignore` if it will be regenerated, OR remove
   the hardcoded pattern entirely and confirm the generated pattern is no longer needed.
6. Document required Vercel env vars.

**This cleanup must happen before Phase 3 (monorepo shell creation).**

**Resolves:** Open Question #4 from the audit.

## Decision 5 — Old Repo Backup Strategy

**Decision: keep both the existing GuideForge and Techsperts repos as independent backups
throughout the migration.**

**Why:** If the monorepo move breaks something unexpectedly, having the original repos
means the rollback is a git clone, not a reconstruction.

**Policy:**
- Do not delete either repo.
- Do not archive either repo until both apps deploy successfully from `apps/guideforge`
  and `apps/techsperts` and have been confirmed stable in production for at least 30 days.
- The new `sonaeko-platform` monorepo becomes the authoritative source of truth only after
  that stability window passes.
- After stability is confirmed, archive (not delete) the old repos on GitHub.

**Resolves:** Open Question #9 (Supabase projects separate), implicitly — both apps keep
their separate Supabase projects; there is no DB merge.

## Decision 6 — Shared Package Extraction Strategy

**Decision: extract shared logic gradually after both apps are moved and verified.**

**Why:** Extracting before both apps are stable in `apps/` creates a moving-target problem
— you are simultaneously restructuring the workspace and changing import paths, making it
hard to determine which change caused a build failure.

**Extraction order:**
1. `docs/` — cross-product architecture docs moved to monorepo root
2. `packages/shared-types` — neutral TypeScript types only; no React; no runtime
3. `packages/domain-templates` — domain scaffold shapes; pure data/constants
4. `packages/core` — neutral lifecycle constants and vocabulary helpers
5. `packages/ai` — prompt contract types and generation schemas first; provider routing later
6. `packages/governance` — neutral status constants, role capability interface, lifecycle types
7. `packages/ai` provider routing — cost-control hooks and mock helpers
8. `packages/design-system` — neutral tokens and status badge primitives last

**Policy:**
- Do not extract runtime database clients (each app's Supabase client stays app-specific).
- Do not extract app UI pages or routing.
- Do not extract auth flows.
- Each shared package extraction is its own dedicated bundle with typechecks and builds.
- Never extract two packages in the same bundle.
- Mark the extraction complete only after typecheck + build passes for both affected apps.

## Decision 7 — Deployment Strategy

**Decision: each product keeps its own Vercel project with separate root directory config.**

**Why:** GuideForge and Techsperts are deployed independently and should continue to be.
A monorepo does not require a shared deployment — Vercel supports per-app root directory
settings for exactly this use case.

**Policy:**
- GuideForge Vercel project root directory: update to `apps/guideforge` after move.
- Techsperts Vercel project root directory: update to `apps/techsperts` after move.
- GuideForge build command remains `next build`. Framework: Next.js.
- Techsperts build command remains `vite build`. Framework: Other/Vite.
- Env vars remain fully app-specific at first. Do not centralize them in the monorepo root.
- No shared Vercel project or shared deployment pipeline during initial move.
- Shared package builds run as part of each app's build (workspace resolution handles this).

**Resolves:** Open Question #10 from the audit.

## Decision 8 — First Monorepo Shell Scope

**Decision: the first monorepo shell creates structure and moves apps as-is. No logic
is extracted and no apps are rewritten during the same session.**

**Allowed in the first shell creation:**
- Create `sonaeko-platform/` repo folder
- Create `apps/` and `packages/` with empty placeholder subfolders
- Create root `package.json` (private, no deps)
- Create `pnpm-workspace.yaml`
- Create root `docs/`
- Copy platform planning docs to root `docs/`
- Copy/move GuideForge to `apps/guideforge/` as-is
- Copy/move Techsperts to `apps/techsperts/` as-is

**Not allowed during the first shell session:**
- Rewrite app imports
- Extract shared code into packages
- Change build frameworks
- Change databases
- Centralize env vars
- Merge UI or force shared styling
- Rename app-internal files
- Touch Supabase schemas or migrations
