# Shared Engine Vocabulary Map

> A durable terminology reference for the sonaeko platform engine.
> Purpose: prevent future AI assistants and contributors from confusing product-specific
> labels with neutral core concepts, or from translating incorrectly between products.
>
> **Rule:** Product labels may differ, but the shared engine concept should remain stable.
> Example: Forged, Verified, and Veteran-reviewed can all label trusted content,
> depending on product and domain.
>
> Pair with `GUIDEFORGE_TECHSPERTS_SHARED_CORE.md` (concept mapping table),
> `PLATFORM_MONOREPO_PLAN.md` (workspace structure), and
> `AI_ASSISTANT_MONOREPO_PROTOCOL.md` (assistant rules).

## Neutral Core Terms

These terms describe the shared engine layer. They do not carry GuideForge or Techsperts
branding. When writing shared packages or cross-product docs, prefer these terms.

| Term | Definition |
|------|------------|
| **Workspace** | The app-level space where a user creates and manages private assets (drafts, plans, networks). Not public-facing until content is published. |
| **Network** | The top-level organizer for a knowledge domain. A named grouping of hubs on a shared subject. May be public or private. |
| **Domain** | The subject area or knowledge category a network covers. Determines which scaffold template, domain profile, and governance rules apply. |
| **Hub** | A mid-level organizer within a network. Groups related collections on a shared sub-topic or device family or problem area. |
| **Collection** | A low-level organizer within a hub. Holds a set of related structured assets (guides, checklists, SOPs, troubleshooting flows). |
| **Asset** | Any structured output: a guide, checklist, SOP, or troubleshooting flow. Has a type, a status, and a lifecycle. |
| **Guide** | A step-by-step walkthrough asset. The primary structured content type. |
| **Checklist** | A multi-step confirmation or procedure asset. Used for pre-flight checks, user prep, or technician steps. |
| **SOP** | Standard Operating Procedure. A formal process asset with a defined owner, trigger, steps, quality check, and escalation path. |
| **Troubleshooting Flow** | A symptom-first, diagnosis-before-fix asset. Structured as: symptom → likely cause → fix → verify → escalate. |
| **Review** | A human evaluation step that moves an asset from draft or provisional toward trusted status. Requires a defined reviewer role. |
| **Verification** | The act of confirming an asset meets the domain standard and is safe to publish. Elevates an asset to trusted status. |
| **Governance Rule** | A domain-level standard that defines verification level, content quality bar, AI policy, and safety requirements for assets in that network. |
| **Trust Badge** | A visual status indicator that shows a reader the asset's current trust level. Label varies by product. |
| **Provisional AI Output** | Any asset generated by an AI call. Always unverified until a human reviewer elevates it. Never auto-published. |
| **Published Output** | An asset that has passed the required review and is visible on the public surface. |

## Asset Status States

These neutral status values apply to all assets in all products. Product labels are mapped
from these states using a product-specific label helper.

| Neutral Status | Meaning |
|----------------|---------|
| `draft` | Created but not submitted for review |
| `provisional` | AI-generated or auto-created; flagged as unverified |
| `in_review` | Submitted to a reviewer; awaiting decision |
| `approved` | Passed review; trust badge applied |
| `deprecated` | Was trusted but no longer current; still viewable but flagged |
| `discarded` | Removed from active use; not public |

## The Core Rule

> Product labels may differ, but the shared engine concept should remain stable.

When a future assistant or contributor reads code in `packages/core`, `packages/ai`, or
`packages/governance`, they should use the **neutral core terms** in that table, not
GuideForge or Techsperts labels.

When a future assistant is working inside `apps/guideforge`, it should use GuideForge
labels in UI copy and product docs, but map them back to neutral types when touching
shared package boundaries.

When a future assistant is working inside `apps/techsperts`, it should use Techsperts
labels in UI copy and product docs, but map them back to neutral types when touching
shared package boundaries.
